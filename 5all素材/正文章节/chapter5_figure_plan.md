# 第五章图表生成计划（v2，根据三方审核修订）

## 目标

为《水面目标精准定位技术报告》第五章生成符合学术规范的全部图表素材。

## 学术规范统一要求

- 图表标题格式：图5-X + 空格 + 标题（宋体小四），表5-X + 空格 + 标题（宋体小四加粗）
- 坐标轴标注：中文标签 + 单位（如"定位误差 (m)"、"定位成功率 (%)"）
- 图例：右上角或最佳位置，不遮挡数据
- 分辨率：300dpi
- 字体：中文 Microsoft YaHei / SimHei，英文 Arial / Times New Roman
- 数据精度：百分比保留1位小数（如82.3%），误差保留1位小数（如28.1m），中位数保留1位小数
- 图像宽度：论文排版适合 2400px 宽（300dpi 下约 20cm）

## 写作口径约束（逐节）

| 节 | 口径要求 | 对图表的影响 |
|---|---------|------------|
| 5.1 | 避免"水面人体目标数据集展示"，表述为"验证低空视觉定位基础链路" | 图5-1标题不含"水面"字样 |
| 5.5 | 不使用"复现优于/低于原论文"，写"低空多视角视觉定位链路测试结果" | 表5-7不出现"复现"列，图5-3不与论文对比 |
| 5.6 | 不把卫星图低精度简单归因为算法失效 | 图5-4解释文字说明参考图质量差异 |
| 5.7 | 样本数少的分组应标注，避免单因子归因 | 图5-5标注"n=22"等样本量，说明耦合关系 |
| 5.8 | 主线集中在Top-N Re-rank、RoMa匹配质量和先验噪声敏感性 | 图5-6聚焦这三个对比维度 |

## 一、B类图表（需重跑实验，核心阻塞）

### 1.1 关键技术约束

- **pkl 不存储原始匹配点**（仅保存 inlier 计数），**必须重跑完整 pipeline** 才能获得匹配可视化
- Baseline.py 的 `if os.path.exists(VG_pkl_path): continue` 会跳过已有 pkl 的样本 → **必须用独立脚本**，不能修改 Baseline.py
- `process_and_save_matches()` 仅绘制内点 → 失败案例需新增 `process_and_save_all_matches()` 绘制全部点
- PnP 结果叠加图需新写函数（在参考地图上标注 pred_loc + truePos）
- RoMa 模型首次加载需额外时间，8个样本串行处理只需加载一次
- config 使用深拷贝，不修改全局 config.yaml
- 场景名均为英文（QZ_SongCity 等），cv2.imwrite 中文路径风险不触发

### 1.4 脚本实现方案

新建独立脚本 `scripts/report/generate_b_type_figures.py`，与已有 `generate_ch5_figures.py` 独立运行。

```python
# 伪代码
# 1. 加载 config（深拷贝，启用 SHOW_RETRIEVAL_RESULT=True）
# 2. 初始化 CAMP 模型（加载一次）
# 3. 初始化 RoMa 模型（加载一次）
# 4. 对每个选定样本：
#    4.1 从 pkl 读取 truePos、路径、参数
#    4.2 加载 UAV 图像（PIL，处理中文路径）
#    4.3 跑 retrieval_all()，保存检索候选图块
#    4.4 跑 Match2Pos_all()：
#        - 保存匹配点对可视化（成功案例：仅内点；失败案例：全部点+仅内点）
#        - 获取 BLH_list、inliers_list
#    4.5 新写函数：在参考地图上标注 pred_loc（红）+ truePos（绿）
#    4.6 拼接为图5-3/图5-4/图5-8的行列格式
# 5. 保存到素材/截图/
```

新增函数：
- `process_and_save_all_matches(Sen_pts, Ref_pts, uav_image, ref_image, save_path)`: 绘制全部匹配点（含outlier，红色；内点绿色）
- `draw_pred_on_refmap(ref_map, pred_loc, truePos, config, region, save_path)`: 在参考地图上标注预测/真值位置

## 三、表格数据修正方案

### 3.1 俯仰角70-90°样本数差异（103 vs 164）

问题：pitch_bucket_stats.csv记录103张，但原始CSV中pitch>=70°有164张。合计426而非487。

原因分析：原始提取脚本可能使用了绝对值比较（`pitch >= 70`）而非范围比较（`70 <= pitch <= 90`），或原始实验有额外过滤条件。

修正方案：
1. 从 results_HIGH.csv 重新统计俯仰角分桶
2. 如果164是正确值，更新表5-9和所有相关图表
3. 在表格中加脚注说明数据来源和统计口径
